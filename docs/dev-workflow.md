# 開発運用フロー

個人開発（知人にテストしてもらいながら進める）を安全かつ抜け漏れなく進めるための、
一連の手順。新しい開発作業に着手する前は、このフローに沿って進める。

## 1. 通常の機能開発

1. `shor-app`フォルダ（`main`ブランチ・開発用Supabase・`IS_DEV=true`）でコードを変更する
2. 画面/挙動に影響する変更は、**実際にブラウザで動かして確認してからコミットする**。
   「コードを読んで動くはず」で済ませない
   - 1人のユーザーの範囲で完結する変更（UI文言、投稿フォームなど）は、
     ブラウザ1つで確認すれば十分
   - 「投稿した人」と「閲覧する人」のように**2人のユーザーが絡む変更**
     （配信ロジック・閲覧枠・現像インタラクションなど）は、
     ブラウザを2つ（例: 通常タブ＋シークレットタブ）開く。
     `localStorage`のuidがタブごとに別になるので、片方で投稿、もう片方で
     閲覧、という一連の流れを**シードデータ無しで自分ひとりで再現できる**
     （`canView()`は自分の投稿を除外するだけなので、この2タブ運用で十分）
3. 変更に対応するdocsを更新する（`CLAUDE.md`の対応表通り）。
   リリース前提条件に関わる変更（環境変数、マイグレーション、外部依存の追加など）
   がある場合は`docs/release-checklist.md`も更新する（後述）
4. `main`にcommit・push

## 2. リリース判断

1. `docs/release-checklist.md`を上から確認する
2. `shor-app-production`フォルダで`git merge main`→push
3. デプロイ後、**必ず自分で本番URL
   （https://euspec-dev.github.io/shor-app/shor.html）を開いてgolden pathを
   一通り確認する**。これが知人に触ってもらう前の最終ゲート
4. 問題があれば、`shor-app-production`で直前の正常なcommitに`git revert`して push

## 3. 知人によるテスト

このアプリにステージング環境は用意しない（別Supabase・別デプロイ先を維持する
コストの方が、個人開発の規模に対して見合わないため）。知人のテストは常に
**本番URL・本番Supabaseに対するソフトローンチ**で行う。

1. 上記「2. リリース判断」の手順で本番にデプロイし、自分で動作確認する
2. 問題なさそうであれば、その知人に一声かけて試してもらう
3. 配信ロジックや閲覧枠など影響の大きい変更の直後は、他の知人が気づく前に
   一通り自分と1人の知人で試すくらいの時間差を意識する
4. 不具合報告が来たら、まず自分で本番URLで再現を試みる。「コードのバグ」か
   「翌日結果など仕様通りの挙動」かをDBを直接いじる前に切り分ける
   （本番DBの読み取りクエリで現状確認するのはOK。書き込みは次項の手順に従う）

## 4. 本番データへの直接介入（サポート対応）

知人のアカウントリセットなど、本番DB・知人のブラウザのlocalStorageを直接
いじる作業は、通常の開発フローとは別枠の運用として扱う。

1. 先に読み取り専用のクエリで現状を確認する（いきなりUPDATE/DELETEしない）
2. 何を・なぜ変更するかを明示し、実行前に必ずユーザー（このドキュメントの
   読み手本人）の確認を取る
3. 実行後、意図した結果になったか確認する
4. 同じ対応が繰り返されるようであれば、その場しのぎのcurl/SQLをやめて、
   再利用できるスクリプトを用意することを検討する

## 5. ロールバック

- 本番で問題が出たら、`shor-app-production`で`git log`を見て直前の正常な
  commitに`git revert`→push
- Supabase側のデータ破損時のバックアップ体制は現状未確認（要検討事項）
